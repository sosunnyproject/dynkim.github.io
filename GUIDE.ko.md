# dynkim.github.io — 프로젝트 가이드 (초보자용)

이 문서는 이 포트폴리오 사이트가 **어떻게 작동하는지**, 그리고 **새 프로젝트를 추가하거나 내용을 수정하려면 어떤 파일을 건드려야 하는지**를 코딩에 익숙하지 않아도 따라갈 수 있게 풀어 쓴 가이드입니다.

---

## 한눈에 보는 이 사이트

- **무엇**: 3D 환경 아티스트의 영문/한글 포트폴리오 웹사이트
- **어디서 작동**: GitHub Pages (GitHub의 무료 웹사이트 호스팅 서비스)
- **어떻게 만들어졌나**: HTML, CSS, JavaScript 만으로 구성된 **정적 사이트**
  - "정적 사이트"란? → 서버나 데이터베이스 없이, 파일들만으로 작동하는 웹사이트. 누구나 페이지를 방문해도 똑같은 내용을 봄.
- **빌드 단계 없음**: 코드를 수정한 뒤 따로 컴파일하거나 변환할 필요 없이, 파일을 저장하고 GitHub에 올리면 끝.

> 💡 추후에는 Notion을 콘텐츠 편집 도구처럼 쓸 수 있게 연결할 예정 — 그렇게 되면 코드를 직접 안 건드리고 Notion에서 글만 고쳐도 사이트가 업데이트됨. 자세한 계획은 [NOTION_CMS_PLAN.md](NOTION_CMS_PLAN.md).

```
dynkim.github.io/

├── game/index.html         ← /game 경로용 특수 페이지

├── i18n.js                 ← 한/영 언어 전환 처리
├── rendering.js            ← 데이터를 받아서 실제 화면을 그리는 코드
├── navigation.js           ← 페이지 이동, 뒤로가기 등 화면 전환 처리
├── routing.js              ← URL과 페이지 연결 (예: #wizard-of-oz-sphere)
├── scroll.js               ← 스크롤할 때 일어나는 효과
├── main.js                 ← 사이트가 처음 켜질 때 실행되는 코드
├── index.css               ← 모든 스타일 (색상, 폰트, 레이아웃)
├── images/                 ← 모든 이미지 파일
├── videos/                 ← 모든 비디오 파일 (.mp4)
├── PDF/                    ← 이력서 PDF 파일들
├── GUIDE.md                ← 영문 가이드
├── GUIDE.ko.md             ← 이 파일 (한글 가이드)
├── NOTION_CMS_PLAN.md      ← Notion 연동 계획서
└── README.md               ← 저장소 소개
```

**파일을 수정해야 할 때 가장 자주 건드리는 곳은 `data.js` 하나입니다.** 나머지는 한번 설정해두면 거의 손댈 일이 없음.

## 핵심 시스템 (어떻게 돌아가는지)

### 1. 한국어/영어 두 언어 지원 (i18n)

> "i18n"이란? → "internationalization" (국제화)의 줄임말. i와 n 사이에 18글자가 있다고 해서 i18n.

사용자에게 보이는 모든 글자는 **두 가지 언어 버전을 함께 저장**합니다. 예시:

```js
name: { en: "Vintage Telephone", ko: "빈티지 전화기" }
```

이 코드를 풀어 읽으면: "이름이라는 정보가 있는데, 영어로는 'Vintage Telephone', 한국어로는 '빈티지 전화기'."

화면을 그릴 때는 `t()`라는 함수가 "지금 사용자가 어느 언어로 보고 있지?"를 확인하고 알맞은 쪽을 골라줍니다. 사용자가 영어 버전을 안 정해줬으면 자동으로 영어로 떨어지도록 안전장치도 있음.

상단 네비게이션의 **EN/KO 토글 버튼**을 누르면 사이트가 즉시 다시 그려지면서 언어가 바뀝니다.

### 2. 같은 사이트, 여러 버전 (Variants)

이 포트폴리오는 **하나의 사이트지만 보는 사람에 따라 첫 화면이 다르게 보이도록** 설계되었습니다. 게임 회사에 지원할 때는 게임 작업이 먼저 보이고, VFX 회사에 지원할 때는 VFX 작업이 먼저 보이는 식.

이를 "variant"(변형)라고 부르고, `data.js`에 `VARIANTS`라는 이름으로 정의되어 있음. 4가지가 있음:

| Variant 이름 | 언제 보이나? | 어떤 작업이 먼저? |
|---|---|---|
| `default` | `dynkim.github.io` (기본 주소) | 균형 있게 섞인 순서 |
| `vfx` | URL에 `vfx`가 포함될 때 | VFX & 애니메이션 작업 |
| `lbe` | URL에 `lbe`가 포함될 때 | LBE (몰입형 체험관) 작업 |
| `game` | URL에 `game`이 포함될 때 | 게임 환경 아트 작업 |

> **LBE 가 뭐야?** → "Location-Based Experience". 오프라인 현장에서 체험하는 전시·공연 같은 것 (Sphere의 The Wizard of Oz 처럼).

각 variant마다 `data.js`에서:
- `row1`, `row2` → **위/아래 줄에 어떤 프로젝트를 어떤 순서로 보여줄지** 직접 정해둠 (자동 정렬이 아니라, 손으로 정한 순서)
- `label1`, `label2` → 각 줄 위에 표시되는 영문/한글 제목

자동으로 어떤 variant인지 알아내는 방법:
1. 도메인 주소(hostname)에 키워드 있는지 확인
2. 경로(path)에 키워드 있는지 확인 (예: `/LBE`)
3. URL 해시(`#lbe` 같은 것)에 키워드 있는지 확인
4. 다 없으면 → 이전에 봤던 것 기억해서 사용

### 3. 프로젝트 데이터 (가장 중요)

`data.js`에 **두 개의 큰 목록**이 있음:

- **`PROJECTS`** → 메인 작업 목록 (개인 프로젝트)
- **`PROJECTS_SECONDARY`** → 회사/스튜디오 작업 목록 (Magnopus 등)

이 둘을 나눈 이유: 홈 화면의 위/아래 두 줄을 명확하게 구분하기 위해, 그리고 URL 주소가 서로 겹치지 않게 하려고.

각 프로젝트는 다음과 같은 정보를 담은 객체(object)로 표현됨:

```js
{
  id: "wizard-of-oz-sphere",          // 고유 식별자 (URL에 사용됨)
  name: { en: "...", ko: "..." },     // 프로젝트 이름
  tag: { en: "...", ko: "..." },      // 작은 태그 (예: "환경", "프롭")
  wip: false,                          // 작업 중인지 여부 (true이면 "WIP" 표시)
  description: { en: "...", ko: "..." }, // 카드에 표시되는 짧은 설명
  year: "2025",                        // 작업 연도
  client: "Sphere Entertainment",      // 클라이언트 (선택)
  externalPartner: "Google, Warner Bros.", // 외부 파트너 (선택)
  status: { en: "Released", ko: "출시됨" }, // 진행 상태
  software: "Unreal 5 · OKO · Maya",   // 사용한 프로그램들
  role: { en: "...", ko: "..." },      // 본인의 역할
  tags: { en: [...], ko: [...] },      // 여러 개의 태그
  thumb: "spwz-01.jpg",                // 홈에 보이는 썸네일 이미지
  thumbVideo: "spwz-thumb.mp4",        // 썸네일을 비디오로 (선택)
  images: [ /* 상세 페이지의 갤러리 */ ]
}
```

> 💡 **id에 대해**: 한 번 정하면 바꾸지 않는 게 좋음. URL이 `dynkim.github.io/#wizard-of-oz-sphere` 같은 형태로 만들어지기 때문에, 변경하면 이미 공유한 링크가 깨짐.

### 4. 갤러리 아이템 종류 (상세 페이지에 들어가는 블록들)

프로젝트 상세 페이지는 위에서 아래로 스크롤하며 보는 형태. `images: [...]` 안에 들어가는 각 항목이 **하나의 블록**이 됨. 6가지 종류:

| 종류 | 어떻게 생겼나? | 필요한 정보 |
|---|---|---|
| **이미지** | 일반 사진 | `src` (이미지 파일명) |
| **비디오** | 자동재생되는 무음 비디오 | `video` (비디오 파일명), `src` (포스터 이미지) |
| **Pair (페어)** | 좌우 나란히 이미지/비디오 2개 | `pair: [{...}, {...}]` |
| **텍스트 (일반)** | 단순 문단 | `text: { en, ko }` |
| **텍스트 (라벨 블록)** | 노란 소제목 + 줄바꿈된 항목들 | `text: { en, ko: { title, items } }` (§5 참고) |
| **Summary (요약)** | 큰 따옴표로 감싸인 강조 박스 | `summary: true` + `text` |
| **링크** | 외부 사이트로 이동하는 버튼/카드 | `link`, `label: { en, ko }` |

선택적으로 추가할 수 있는 옵션:
- `caption: { en, ko }` → 이미지/비디오 아래 작은 캡션
- `narrow: true` → 비디오 폭을 좁게 표시
- `aspect: "16/9"` → 링크 카드의 가로세로 비율

### 5. 라벨 블록 — 긴 글을 깔끔하게 정리하는 방법 (신규)

긴 한국어 설명을 **소제목 + 줄바꿈된 문장 리스트**로 깔끔하게 정리하기 위한 방식. 예를 들면 이렇게 보임:

```
                  프로젝트 배경                ← 노란색, 가운데
                  ─────────────────

  Sphere Entertainment, Google, Warner Bros., 
  Magnopus는 Sphere에서 〈The Wizard of Oz〉를...

  공연의 시뮬레이션·리허설·비주얼 테스트를 수행할 
  수 있는 디지털 공간 제작을 추진했다.
```

코드로는 이렇게 작성:

```js
{
  text: {
    en: "In August 2025, Sphere Entertainment presented...",
    ko: {
      title: "프로젝트 배경",
      items: [
        "Sphere Entertainment, Google, Warner Bros., Magnopus는...",
        "공연의 시뮬레이션·리허설·비주얼 테스트를..."
      ],
      bulleted: false     // summary 블록일 때만 true (불릿 표시)
    }
  }
}
```

- `title` → 노란 소제목 (예: "Task", "Action", "Result")
- `items` → 한 줄에 한 문장씩 들어갈 내용들 (불릿 없음, 줄바꿈만)
- `bulleted` → summary 박스에서만 `true`로 설정 (따옴표 + 불릿 둘 다 표시됨)

> 💡 **현재 적용된 곳**: `wizard-of-oz-sphere` 프로젝트만. 다른 프로젝트들도 점차 이 형식으로 옮길 예정.

해당 스타일링 CSS는 `index.css` 하단에 `.labeled-block`, `.labeled-title`, `.labeled-items` 클래스로 정의됨.

## /LBE, /game 페이지가 어떻게 작동하나?

`dynkim.github.io/LBE`로 접속하면 GitHub Pages가 `LBE/index.html` 파일을 보여줌. 이 파일은 **메인 `index.html`을 그대로 복사한 것**에 한 줄만 추가:

```html
<base href="/">
```

> **이 한 줄이 왜 필요해?** → HTML 파일이 하위 폴더에 들어있으면, 브라우저가 이미지 경로를 잘못 찾을 수 있음. `<base href="/">`는 "모든 경로를 사이트 루트에서부터 찾아라"라고 알려줌. 이 한 줄 덕분에 `images/sg-01.jpg` 같은 경로가 정상 작동.

JavaScript는 동일한 코드를 사용하지만, URL이 `/LBE`인 걸 보고 LBE variant로 자동 전환됨.

### LBE/game 페이지를 동기화하는 법

`index.html`을 수정했다면, **같은 변경을 `LBE/index.html`과 `game/index.html`에도 똑같이 적용**해야 함. 유일하게 다른 부분은 `<base>` 태그 하나뿐.

이 작업이 번거로워지면 → 장기적으로는 Notion 빌드 파이프라인으로 옮겨서 한 곳에서만 관리하면 됨.

## 새 프로젝트를 추가하는 방법

### 1단계: 이미지/비디오 파일 준비

이미지는 `images/` 폴더에, 비디오는 `videos/` 폴더에 넣음. 파일명에 일관된 접두어를 사용하면 관리하기 좋음 (예: `myproj-01.jpg`, `myproj-02.jpg`...).

### 2단계: `data.js`에 프로젝트 정보 추가

`PROJECTS` (개인 작업)나 `PROJECTS_SECONDARY` (회사 작업) 배열 어딘가에 새 프로젝트 객체를 추가. 가장 비슷한 기존 프로젝트를 복사한 다음 내용만 바꾸는 게 가장 쉬움.

### 3단계: variant 순서에 추가

`data.js`의 `VARIANTS` 부분에서, 이 프로젝트가 나타나야 할 variant 마다 `row1` 또는 `row2` 배열에 새 프로젝트의 `id`(slug)를 넣음. 순서가 곧 화면에 보이는 순서.

### 4단계: 갤러리 채우기

상세 페이지에 들어갈 이미지, 비디오, 텍스트 블록들을 `images: [...]` 배열에 채움. 위 §4의 6가지 종류 중에서 골라 사용.

## 일반 텍스트를 라벨 블록으로 바꾸는 방법

지금 이런 식으로 작성된 한국어 텍스트가 있다면:

```js
ko: "프로젝트 배경: 본 프로젝트는 LA28 올림픽... 첫 문장입니다. 두 번째 문장입니다."
```

이렇게 바꾸면 됩니다:

**1단계**: 첫 번째 콜론(`:`)을 기준으로 라벨과 본문을 분리.
- 라벨: `프로젝트 배경`
- 본문: `본 프로젝트는 LA28 올림픽... 첫 문장입니다. 두 번째 문장입니다.`

**2단계**: 본문을 마침표 + 공백 기준으로 문장별로 자름.
- `본 프로젝트는 LA28 올림픽... 첫 문장입니다.`
- `두 번째 문장입니다.`

**3단계**: `text.ko`를 객체 형태로 교체:

```js
text: {
  en: "...기존 영어 문단 그대로 유지...",
  ko: {
    title: "프로젝트 배경",
    items: [
      "본 프로젝트는 LA28 올림픽... 첫 문장입니다.",
      "두 번째 문장입니다."
    ]
  }
}
```

영어 쪽은 그대로 두면 됨 — 렌더링 코드가 한국어 쪽만 객체인 걸 자동으로 알아채고 라벨 블록으로 표시함.

## 로컬에서 미리보기 하는 방법

이 사이트는 빌드 도구가 없어서, 그냥 브라우저에서 열기만 하면 됨. 하지만 어떤 기능들(상대 경로, 해시 라우팅 등)은 **로컬 서버**가 필요할 수 있음.

터미널에서 프로젝트 폴더로 이동한 뒤:

```bash
python3 -m http.server 8000
```

그리고 브라우저에서 `http://localhost:8000` 열기.

다른 variant를 테스트하려면:
- `http://localhost:8000/#lbe`
- `http://localhost:8000/#game`
- `http://localhost:8000/#vfx`

해시(`#`) 뒤에 variant 이름을 붙이면 됨. 하위 폴더 복사본을 만들 필요 없음.

## GitHub에 올려서 배포하기

수정을 끝냈다면, 변경사항을 GitHub에 푸시:

```bash
git add -A
git commit -m "변경 내용 한 줄 설명"
git push
```

GitHub Pages가 자동으로 약 1분 안에 사이트를 다시 빌드해서 반영합니다.

> ⚠️ **주의**: `git add -A`는 모든 변경 파일을 한꺼번에 올림. 비밀 정보(API 키, 비밀번호 등)가 든 파일이 있다면 따로 `.gitignore`에 등록해두는 게 좋음.

## 문제가 생기거나 헷갈릴 때

- **이미지가 안 보임** → 파일명 오타 확인 (대소문자 구분), `images/` 폴더에 실제로 파일이 있는지 확인
- **새 프로젝트가 홈에 안 나타남** → `VARIANTS`의 `row1`/`row2` 배열에 id를 넣었는지 확인
- **한국어 토글 시 영어로 떨어짐** → `name`, `description` 등에 `ko:` 필드를 빼먹었는지 확인 (영어 폴백이 작동 중인 것)
- **상세 페이지에서 텍스트가 깨져 보임** → JSON 형식 오류일 가능성. `data.js`를 저장한 뒤 브라우저 개발자 도구(F12) → Console 탭에서 빨간 에러 메시지 확인
